# Webcrawla — BYOK AI Overviews for Startpage

**Date:** 2026-07-01
**Status:** Approved by user

## Purpose

A Chrome MV3 browser extension (for Vivaldi, which runs Chrome extensions natively) that injects an AI-generated answer card at the top of Startpage search results — emulating Google's Gemini AI Overviews, but bring-your-own-key: the user supplies any OpenAI-compatible endpoint, API key, and model name. Initial target: Ollama Cloud (`https://ollama.com/v1`) with GLM 5.2.

The summary is grounded in a real crawl of the result pages (the model reads actual page content, not just search snippets), with the model able to fetch additional pages mid-generation when it needs more.

## User experience

1. User searches on Startpage as normal; results render untouched.
2. If the query looks like a question/research intent (smart trigger), a card injects above the first organic result showing a "Reading sources…" loading state. For other queries, the card renders collapsed with a one-click "Summarize" button instead.
3. The answer streams in word-by-word as sanitized markdown with inline numbered citations (`[1]`, `[2]`) rendered as clickable superscript links, followed by a favicon strip of sources.
4. Card footer: "generated by <model> via <endpoint host>", a retry button, and a collapse toggle.
5. If no API key is configured, the card is a setup prompt linking to the options page.

## Architecture

Four components:

### 1. Content script (`startpage.com` result pages)

- Scrapes the rendered SERP DOM: for each organic result, extract title, URL, snippet. All selectors live in one module (`serp-selectors.ts`) with a generic fallback (heading + `a[href]` detection) so Startpage markup changes degrade gracefully.
- Injects the answer panel inside a **shadow DOM** root above the first result (style isolation both directions).
- Runs the trigger heuristic locally: question words, query length, absence of navigational patterns (e.g. "login", bare domains) → auto-run; otherwise manual button.
- Opens a long-lived `chrome.runtime` port to the background worker; sends `{query, results[]}`, receives streaming events (`status`, `token`, `sources`, `done`, `error`).
- Re-runs on in-page navigation (new search from the results page).

### 2. Background service worker (agent orchestrator)

- **Hybrid race:** on receiving a job, immediately fires parallel fetches of the top 5 result URLs (per-page timeout ~4s; slow/failed pages skipped and marked). Extracted content becomes numbered sources in the prompt.
- Calls the configured OpenAI-compatible `/chat/completions` endpoint with **streaming enabled and tool calling**.
- One tool exposed to the model: `fetch_page(url)` — fetch + extract any additional URL (another result, a link found in a fetched page). Caps: **max 3 tool rounds, max 8 total pages fetched**, hard **45s** overall budget.
- Prompt instructs the model to answer concisely with `[n]` citations matching the numbered source list, and to say so when sources are insufficient.
- Streams tokens back over the port as they arrive.

### 3. Offscreen document (HTML → text extraction)

MV3 service workers have no DOM, so extraction runs in a `chrome.offscreen` page: parse fetched HTML with `DOMParser`, run `@mozilla/readability`, return clean article text truncated to a per-page character budget (default ~8,000 chars ≈ 2k tokens).

### 4. Options page

Settings stored in `chrome.storage.local` (never `sync` — the API key stays on-device and is only ever sent to the user-configured endpoint):

- Endpoint base URL (default `https://ollama.com/v1`)
- API key
- Model name
- Trigger mode: smart (default) / always / manual
- Max prefetch pages (default 5), per-page content budget
- Optional system-prompt override

## Data flow

```
results page load
  → content script scrapes SERP + evaluates trigger
  → port message {query, results[]} to background
  → background: parallel prefetch top 5 → offscreen extraction
  → streaming chat/completions call (sources in prompt, fetch_page tool available)
  → [optional tool rounds: model requests more pages]
  → token stream → content script renders markdown into panel
```

## Manifest / permissions

- `permissions`: `storage`, `offscreen`
- `host_permissions`: `https://*.startpage.com/*` (content script) + `<all_urls>` (background fetch of arbitrary result pages and the LLM endpoint). Cross-origin fetches from the background worker bypass CORS via host permissions — this is why an extension (not a userscript) is the right form factor.

## Security & privacy

- API key: local storage only, sent only to the configured endpoint.
- Model output is rendered through a small sanitizing markdown renderer — model text never becomes raw HTML.
- Fetched page content goes only to the user's own endpoint.
- Fetch sizes capped; non-HTML responses skipped.

## Error handling

| Failure | Behavior |
|---|---|
| No API key | Setup card linking to options |
| Page fetch fails/times out | Skip, mark source as unavailable, continue |
| LLM/network error | Readable error in card + retry button |
| Zero pages extractable | Fall back to snippet-only summary, labeled as such |
| Overall budget exceeded (45s) | Abort remaining work, finish with what streamed |

## Stack

- TypeScript, bundled with esbuild (no framework; vanilla DOM in shadow root)
- `@mozilla/readability` — the only substantial runtime dependency
- Tiny sanitizing markdown renderer (own module or micro-dep)
- Tests: vitest for trigger heuristic, SERP scraper (fixture HTML), citation parser, prompt assembly. E2E: load unpacked in Vivaldi against live Startpage.

## Known risks

- **Startpage DOM drift:** selectors are not a public API. Mitigated by the single-module selector layer + generic fallback; breakage degrades to the manual button with a "couldn't read results" note rather than failing silently.
- **Bot defenses on crawled sites:** some pages will block extension fetches (403/challenge). Handled as fetch failures; the model is told which sources were unavailable.

## Out of scope (v1)

- Follow-up chat in the panel (planned v2)
- Other search engines (selector layer is designed to make this easy later)
- `web_search` tool for the model (fetch_page only in v1)
