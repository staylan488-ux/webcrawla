# Webcrawla v2 — Follow-up Chat

**Date:** 2026-07-01
**Status:** Approved by user
**Builds on:** `2026-07-01-webcrawla-ai-overview-design.md` (v0.1.0, shipped)

## Purpose

Expand a finished AI overview card into an inline chat: the user asks follow-up questions about the summary and gets streamed, citation-grounded answers from the same model. Follow-ups keep the `fetch_page` tool — the model may crawl additional pages per question (same caps as a summary run: 8 pages, 3 tool rounds, 45 s — applied per exchange).

## UX

1. When a summary finishes, the card footer shows **"💬 Ask a follow-up"** next to Regenerate.
2. Clicking reveals a chat area under the answer: a text input pinned at the bottom; Enter or a send button submits.
3. Each exchange renders: the user's question as a compact quoted line, then the streamed reply as a markdown block with citation superscripts (same sanitizing renderer).
4. New pages fetched during a follow-up extend the favicon source strip; citation indices continue across the conversation (`[6]`, `[7]`, …).
5. Input is disabled with a shimmer while a reply streams.
6. **Regenerate** clears the conversation (deletes stored transcript) and reruns the summary fresh.
7. Reloading/re-visiting the same search restores the summary and full chat from storage instead of re-running; Regenerate remains the escape hatch.
8. Follow-up errors render as an error line in the chat with a retry button. If the stored conversation is gone (evicted or browser restarted), the panel shows "Conversation expired — regenerate to start fresh."

## Conversation state

- Stored in **`chrome.storage.session`** (cleared when the browser closes; survives MV3 service-worker restarts and page reloads).
- Keyed by **jobId** (`crypto.randomUUID()`, generated by the content script per summary run): `wc:job:<jobId>` → `{ messages, sources, display, updatedAt }`.
  - `messages`: full ChatMessage transcript (system, user w/ source content, assistant turns, tool results).
  - `sources`: the ExtractedSource array (indices stable across the conversation).
  - `display`: lightweight render transcript `{ role: 'user' | 'assistant', markdown: string }[]` — element 0 is the summary — used to restore the card without re-running.
- Query→job index `wc:query:<normalized query>` → jobId enables restore on reload.
- **Eviction:** keep the 5 most recent jobs; prune older ones (and their query-index entries) on every save.

## Protocol changes (`src/shared/types.ts`)

- `JobRequest` (`type: 'run'`) gains `jobId: string`.
- New request: `{ type: 'followup'; jobId: string; question: string }`.
- `StreamEvent` unchanged — events describe the current exchange only.

## Background changes

- **`src/background/transcript.ts` (new):** load/save/delete/prune conversation records in `chrome.storage.session`; owns the eviction policy and the query index.
- **`src/background/agent.ts`:** extract the streaming tool loop into a reusable core (`runTurns(messages, sources, settings, deps, emit, budget)`). `runAgent` = prefetch + build initial messages + `runTurns` + save transcript. New `runFollowup(jobId, question, settings, deps, emit)` = load transcript (missing → `error` "Conversation expired — regenerate to start fresh."), append user question, `runTurns`, save back. Page/round/time caps reset per exchange; `sources` continues (dedup already prevents re-fetching cited pages — a deduped URL responds "Already fetched as [n]").
- **`src/background/index.ts`:** route `followup` messages; keepalive interval wraps both job kinds.

## Content/panel changes

- **`src/content/index.ts`:** generate `jobId` per run; on init, ask the background whether a conversation exists for the current query via a one-shot runtime message `{ target: 'background', kind: 'get-conversation', query }` → `{ jobId, display, sources } | null` (content scripts cannot read `chrome.storage.session` directly — it is trusted-contexts-only by default) — if non-null, restore instead of auto-running; `startFollowup(question)` opens a fresh port per exchange (same pattern as `startJob`, incl. disconnect recovery).
- **`src/content/panel.ts`:** replace the single markdown accumulator with per-exchange render targets: each exchange owns a DOM block and its accumulator (the summary is exchange 0). Panel API additions: `enableChat(onAsk)`, `addUserMessage(text)`, `beginExchange()` / per-exchange `appendToken`/`finish`/error, `restore(display, sources)`. Citation map shared panel-wide and updated by `sources` events.

## Error handling

| Failure | Behavior |
|---|---|
| Follow-up LLM/network error | Error line in chat + retry button (re-sends the same question) |
| Stored conversation missing | "Conversation expired — regenerate to start fresh." |
| Port disconnect mid-follow-up | Same recovery as summary runs (error + retry) |
| storage.session write failure | Non-fatal: answer still renders; conversation just won't survive restarts |

## Out of scope

- Persistent (cross-browser-session) history and any history-browsing UI.
- Editing/deleting individual messages; branching conversations.
- Follow-up suggestions/chips.

## Testing

- Unit: transcript store (mocked `chrome.storage.session` — save/load/prune/index), `runFollowup` (existing fake-deps pattern: appends question, reuses sources, caps enforced, expired-job error), panel chat (enableChat/addUserMessage/exchange isolation — a second exchange never mutates the first block; restore path renders display transcript).
- Existing 68 tests must keep passing; `runAgent` behavior is unchanged from the outside.
- Manual E2E in Vivaldi: ask a follow-up that needs no crawl (fast path), one that does (watch sources grow), reload the page and confirm restore, regenerate clears.
